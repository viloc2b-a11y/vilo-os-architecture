\# Visit Runtime Engine



Status: Draft



\---



\# Objective



Define the executable runtime engine that orchestrates subject visits, procedure execution, operational state, source capture, workflow progression, and runtime lineage.



The Visit Runtime Engine is the operational heart of Vilo OS.



\---



\# Core Principle



Visits are executable operational states, not static calendar entries.



\---



\# Runtime Chain



Protocol

→ Visit Definition

→ Visit Instance

→ Procedure Execution

→ Operational Events

→ Computed Visit State

→ Governance Signals

→ Financial Signals



\---



\# Runtime Responsibilities



The Visit Runtime Engine is responsible for:



\- visit orchestration

\- visit readiness

\- procedure sequencing

\- conditional branching

\- execution tracking

\- runtime validations

\- workflow progression

\- source linkage

\- longitudinal continuity

\- operational event emission



\---



\# Visit Runtime Model



\## Visit Definition



Defines:

\- protocol visit

\- required procedures

\- windows

\- dependencies

\- conditional rules

\- off-site eligibility

\- PK rules

\- safety rules



\---



\## Visit Instance



Represents:

\- actual subject visit execution

\- current operational state

\- procedure completion

\- deviations

\- signatures

\- runtime lineage



\---



\# Visit States



\## Core States



\- Scheduled

\- Pending Readiness

\- Ready

\- In Progress

\- Partially Complete

\- Awaiting Review

\- Locked

\- Signed

\- Closed

\- Deviated

\- Cancelled



\---



\# Procedure Runtime



Each procedure must support:



\- independent execution

\- execution timestamps

\- performer attribution

\- source linkage

\- deviation linkage

\- workflow dependencies

\- conditional execution

\- financial mapping



\---



\# Conditional Runtime



The engine must support:



\- conditional procedures

\- triggered workflows

\- safety escalation

\- PK branching

\- adrenal workflows

\- HIT workflows

\- off-site workflows

\- unscheduled visit generation

\- amendment-aware execution



\---



\# Visit Readiness Engine



The system computes readiness from:



\- required procedures

\- pending labs

\- unresolved deviations

\- safety holds

\- missing source

\- missing signatures

\- protocol dependencies

\- randomization status



The coordinator should never manually determine readiness.



\---



\# Operational Event Model



Every meaningful action emits immutable events.



Examples:



\- visit\_started

\- procedure\_completed

\- source\_signed

\- deviation\_created

\- safety\_triggered

\- workflow\_blocked

\- visit\_locked

\# UX Requirements

\# Explicit Non-Goals



The runtime engine must NOT become:



\- a static EDC

\- a generic form engine

\- a scheduling-only system

\- a reporting dashboard

\- a sponsor oversight tool



\---



\# Final Rule



If runtime complexity becomes visible to the coordinator, the architecture has failed.
