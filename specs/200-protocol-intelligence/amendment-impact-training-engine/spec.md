\# Amendment Impact and Training Engine



Status: Draft



\---



\# Objective



Define the architecture for protocol amendment impact analysis, runtime delta computation, retraining workflows, operational burden calculation, financial impact, and amendment-aware execution continuity.



\---



\# Core Principle



An amendment is not a document update.



An amendment is a runtime mutation with operational, compliance, training, and financial consequences.



\---



\# Responsibilities



The engine must support:



\- amendment intake

\- protocol version comparison

\- visit matrix delta detection

\- procedure delta detection

\- source template delta detection

\- workflow rule delta detection

\- safety rule delta detection

\- training requirement generation

\- financial impact computation

\- runtime version preservation



\---



\# Amendment Delta Categories



The system must detect:



\- new visits

\- removed visits

\- changed visit windows

\- new procedures

\- removed procedures

\- changed procedure timing

\- new safety workflows

\- changed eligibility criteria

\- changed source requirements

\- changed signature requirements

\- changed financial billables

\- changed training requirements



\---



\# Runtime Versioning



Each amendment must generate:



\- new protocol version

\- new runtime version

\- amendment delta record

\- affected subject list

\- affected visit list

\- affected procedure list

\- affected source list

\- affected financial items



\---



\# Training Engine



The system must generate:



\- required retraining tasks

\- role-specific training requirements

\- training completion tracking

\- amendment acknowledgment

\- PI/Sub-I/coordinator training evidence

\- inspection-ready training lineage



\---



\# Operational Impact



The engine must compute:



\- added coordinator workload

\- new visit burden

\- new source burden

\- new safety burden

\- new monitoring burden

\- new financial burden

\- new deviation risk



\---



\# Financial Impact



The engine must produce:



\- amendment budget delta

\- new billable requirements

\- uncompensated work signals

\- negotiation evidence

\- sponsor-facing justification



\---



\# Governance Integration



Amendment workflows must generate:



\- audit lineage

\- protocol version lineage

\- retraining evidence

\- operational impact evidence

\- inspection-ready amendment timeline



\---



\# UX Requirements



Users must see:



\- what changed

\- who is affected

\- what must be retrained

\- what visits/procedures changed

\- what source changed

\- what financial terms need review



\---



\# MVP Scope



MVP must support:



\- amendment version record

\- manual delta entry

\- affected subject/visit/procedure tracking

\- retraining task generation

\- financial impact placeholder

\- immutable amendment events



\---



\# Deferred Scope



Deferred:



\- automated full protocol redline parsing

\- autonomous amendment interpretation

\- sponsor negotiation automation

\- advanced ML amendment risk prediction



\---



\# Explicit Non-Goals



The engine is NOT:



\- document upload only

\- static amendment archive

\- generic training tracker

\- sponsor amendment portal



\---



\# Final Rule



If an amendment changes execution but does not change runtime logic, the architecture has failed.
