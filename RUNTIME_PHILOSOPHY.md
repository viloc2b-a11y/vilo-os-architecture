\# Vilo OS Runtime Philosophy



Status: Active



\---



\# Purpose



Define the operational DNA of Vilo OS.



This document governs how all architecture, product decisions, implementation tasks, UX flows, and agent outputs should be evaluated.



\---



\# Core Thesis



Vilo OS is an execution-native clinical research operating system.



Its purpose is to reduce cognitive load for research site staff while converting protocol complexity into guided, traceable, financially-aware runtime execution.



\---



\# Central Principle



The coordinator should not carry protocol complexity cognitively.



The system orchestrates execution.



\---



\# What Runtime Means



Runtime means the system continuously understands:



\- what protocol version is active

\- what visit is expected

\- what procedures are required

\- what conditions change execution

\- what safety items carry forward

\- what source is required

\- what signatures are missing

\- what work has been completed

\- what revenue has been earned

\- what risks are emerging

\- what evidence exists



\---



\# Runtime Is Not



Runtime is not:



\- static forms

\- dashboards

\- status labels

\- file storage

\- generic task lists

\- sponsor reports

\- AI chat outputs



\---



\# Execution Truth



Execution truth must derive from:



\- operational events

\- protocol graph

\- procedure completion

\- source evidence

\- actor attribution

\- timestamps

\- immutable lineage



\---



\# Coordinator Cognitive Offload



The system must eliminate the need for coordinators to manually remember:



\- protocol dependencies

\- visit windows

\- unresolved safety items

\- source requirements

\- PI signature requirements

\- billable procedures

\- query burden

\- amendment changes

\- follow-up obligations



\---



\# System Responsibilities



The system must continuously compute:



\- readiness

\- blockers

\- risk

\- workload

\- compliance

\- revenue

\- missing evidence

\- next action



\---



\# Human Responsibilities



Humans must:



\- make clinical judgments

\- govern final decisions

\- approve protocol publications

\- sign required records

\- resolve ambiguous findings

\- validate AI-assisted outputs



\---



\# AI Philosophy



AI may assist with:



\- extraction

\- summarization

\- retrieval

\- signal clustering

\- explanation

\- draft recommendations



AI must never replace:



\- PI judgment

\- clinical assessment

\- compliance accountability

\- source truth

\- audit lineage

\- financial authority



\---



\# Governance Philosophy



Governance must be produced by correct runtime.



Not by parallel manual QA systems.



\---



\# Financial Philosophy



Finance must be derived from operational truth.



Not from static budget spreadsheets alone.



\---



\# UX Philosophy



The best UX is not more screens.



The best UX is fewer decisions, fewer clicks, fewer forgotten tasks, and more guided execution.



\---



\# Site-First Rule



Vilo OS is for sites first.



Sponsors, CROs, monitors, and enterprise reporting are downstream beneficiaries, not the primary design center.



\---



\# Anti-Complexity Rule



Reject anything that makes the system feel like:



\- SAP for clinical research

\- sponsor command center first

\- generic CTMS clone

\- AI wrapper

\- bloated configuration engine



\---



\# Final Test



Every feature must pass this test:



Does it help the site execute protocol work faster, safer, cleaner, with better traceability and better revenue capture?



If not, it does not belong in the MVP.
